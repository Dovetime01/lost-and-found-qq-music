# 失物招领处完整流程增量实现设计

## 目标与原则

以《失物招领处_完整实现方案.md》为唯一产品目标，在保留现有可运行体验的前提下，把当前通用分析流程逐步替换为票根预取、媒体准备、多模态分析、识曲/降级、认领单、五首归途电台、总文案和试听过渡页。实施严格采用最小可验证单元；每个单元独立测试通过后才接入下一个单元。

所有外部服务只在 Next.js 服务端调用。真实密钥仅存在于 `.env.local`。QQ 音乐专用接口若无权限、超时、返回空结果或协议不兼容，统一进入无感降级，不向用户展示技术错误。

## 架构边界

### 1. 会话编排状态

新增一个前端会话状态对象保存当前流程中间结果：演出信息、艺人热门曲缓存、媒体素材状态、多模态分析结果、识曲结果或替代锚点、认领单、五首电台、总文案和各任务就绪状态。该对象只保存本次页面会话，不引入数据库。

页面只负责触发任务和展示状态。外部请求、结果标准化、降级选择、歌曲去重和五首角色分配分别放在独立模块中，避免继续扩大 `page.tsx` 和通用 `aiAnalysis.ts`。

### 2. 外部适配器

- 票根：沿用已完成的百度 OCR + Ark 结构化提取。
- QQ 音乐：扩展现有签名与标准化逻辑，分别封装歌手查询、热门搜索、听歌识曲、相似歌曲和 `music_skill`。所有适配器返回统一 `MusicTrack`，不得把供应商原始响应泄漏给页面。
- 豆包：素材分析、认领单和电台总文案各自使用独立 prompt 与输出类型。音频理解先用最小真实请求验证官方字段；Responses 无法验证时采用 Chat Completions 多模态结构。
- 音频：媒体准备模块负责中段视频帧、单声道压缩音频、原始录音识曲片段和视频音轨。MP3 编码优先采用体积受控、浏览器可用的编码器；若目标浏览器无法编码，则回退到浏览器可生成的单声道压缩格式，并由服务端按 MIME 正确转发，不伪报为 MP3。

## 增量实施单元

1. OCR 后艺人热门曲预取：新增后端预取入口、歌手/热门曲适配器、会话缓存状态和失败降级测试。
2. 视频中段抽帧：提取纯工具，定位 50% 时长，JPEG 质量 0.8，覆盖无时长/解码失败。
3. 音频素材模型：EvidenceArtifact 增加受控音频 Data URL、MIME、原始识曲片段和视频音轨字段；先只建立数据边界和大小校验。
4. 音频处理：单声道压缩、视频音轨提取与录音混合并行执行；保留原始录音前 8-10 秒用于识曲。
5. 单次豆包多模态：图片、压缩音频和文字一次提交，严格输出照片氛围、声音情绪和固定词库单一标签。
6. 识曲与静默降级：低置信度、超时或空结果统一选择热门曲第一候选，并记录内部 `anchorSource`，前端不显示失败。
7. 专用认领单：等待歌曲锚点和多模态结果，生成遗失物名称、发现地点、3-4 句感想、1-3 个标签与 1-10 强度；替代锚点使用不声称精确识曲的发现地点。
8. 五首电台：并行获取相似曲、热门曲和情绪曲，纯函数完成去重、角色分配与第 4 首筛选；任一路失败使用缓存/本地曲库补位。
9. 电台总文案：五首歌曲就绪后生成 50-80 字总介绍和恰好五条 10-20 字推荐语。
10. 试听过渡页：保留搜索动画，随后展示复用黑胶组件的独立页面；支持处理中、全部就绪、20 秒部分就绪三态，不设置强制停留时间。
11. 集成收口：认领单、路线页和播放页消费新数据结构；删除已无调用方的重复通用逻辑，保留必要的本地兜底。

## 数据流与并行

票根成功后立即启动热门曲预取，但不阻塞用户进入素材页。素材提交时，同时启动媒体处理、听歌识曲和豆包多模态分析。锚点确定后，认领单生成与可执行的电台歌曲任务并行。五首歌全部或降级补齐后生成总文案。

搜索动画只承担短过场；随后无条件进入试听页。20 秒计时从试听页挂载开始。全部结果就绪时按钮立即变为可用；20 秒仍未完成时允许用已完成结果与本地补位继续。

## 错误与降级

- 所有外部调用设独立超时并返回可区分的内部状态；页面只看到诗意、非技术性的文案。
- 识曲失败不写具体歌曲副歌时间点；热门曲锚点失败则使用本地同艺人或情绪曲库。
- 多模态失败使用已有图片描述、音频特征和文字规则生成三项结果。
- 电台任何角色缺失时按“同艺人缓存 → 相似/搜索剩余 → 本地曲库”顺序补位，并保证歌曲不重复。
- 20 秒兜底只放宽页面进入条件，不取消仍在后台执行的任务。

## 测试与验收

每个单元执行 RED-GREEN-REFACTOR。适配器测试固定验证请求命令、签名输入、超时、响应标准化与失败回退；纯编排测试验证并行触发、五首角色、去重和锚点措辞；组件测试或浏览器检查验证抽帧位置、试听页三态和流程跳转。

每个单元完成后运行对应测试；每个阶段结束运行全量单测。最终必须通过生产构建，并用真实百度/Ark和可用的 QQ 权限做冒烟测试。无法获得权限的专用 QQ 接口必须明确记录为“适配器和降级已验证，真实供应商成功响应未验证”，不得伪称已真实打通。

## 非目标

- 不在本次方案中补建 Supabase 公共墙。
- 不新增账号体系或跨设备会话缓存。
- 不为了音频编码引入完整媒体服务器；只采用满足当前浏览器 MVP 的最小方案。
- 不删除现有档案柜、失物墙或结尾页功能。
