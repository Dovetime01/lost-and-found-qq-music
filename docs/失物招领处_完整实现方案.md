## 一、整体数据流时序图（谁在什么时候发生）

```
第2页：上传票根
  → 触发OCR识别 → 拿到【艺人名/演出名/日期/城市/场馆】
  → 立刻后台并行发起：QQ音乐"搜歌手" + "按艺人搜热门曲目"（为归途电台③提前准备，此时用户还在往下操作，无感知）

第3页：上传照片/视频/录音/文字
  → 照片：保留原图，作为素材之一
  → 视频：拆成两部分——
       视觉部分：前端抽帧（截取40%-60%时长处的一帧），当成"照片"素材
       音频部分：提取视频音轨，和用户单独录制的跟唱音频归为同一类"音频素材"
  → 录音：先压缩（转为64kbps左右单声道mp3），和视频抽取出的音轨（如果有）合并为同一份音频素材；
         同时这段（未压缩前的）原始录音前8-10秒仍需单独送去"听歌识曲"API识别当晚具体唱的歌
  → 文字：如果用户写了感受，直接留存

第4页：搜索动画（保留） + 新增"先听一首歌"过渡页
  → 原来的搜索动画继续保留，作为极短暂的过场衔接
  → 紧接着进入"先听一首歌"页面（详见第五部分），用户在这里有一段可以停留的、有质感的等待体验
  → 此时后台并行处理：
     A. LLM调用一：把照片素材+音频素材+文字**合并成一次豆包多模态调用**，
        一次性输出【照片氛围描述】+【声音情绪描述】+【情绪标签】三项结果
     B. API①听歌识曲结果返回后，触发API②相似单曲推荐
     C. API③（艺人热门曲目）此时早已在第2页跑完，直接读缓存
     D. API⑤（SearchSong意图，用情绪标签匹配收尾曲）等A完成后即可触发
     E. API④（许久未听）：从②的结果里选一首非当前识别曲目的同艺人早期作品
     F. 如果①听歌识曲失败/未识别出结果，走"备选方案"（详见第三部分）
  → LLM调用二：认领单文案生成（需要等①或其备选方案确定"具体歌名"后才能进行）
  → LLM调用三：归途电台总文案（需要等②③④⑤四首歌全部就绪后才能进行）

第5-6页：展示结果
  → 如果第4页等待体验设计合理，用户几乎感受不到延迟，展示的是早就算好的结果
```

## 二、认领单字段 —— 需要的原始数据 + LLM口令

### 认领单最终要生成的字段
```json
{
  "遗失物名称": "诗意短语，8-16字",
  "发现地点": "《歌曲名》+ 时间点描述",
  "失而复得的感想": "3-4句第一人称文案",
  "情绪标签": ["从固定词库中选1-3个"],
  "情绪强度": "1-10整数"
}
```

### 第一层：素材分析（合并为一次豆包多模态调用）

**输入**：
- 照片（或视频抽帧后的图片）—— 直接传原图
- 音频（用户录音+视频音轨合并后，压缩处理过的版本）—— 转base64传入
- 用户文字（如果有）

**调用豆包（doubao-seed-2.0-lite，支持图像+音频+文本统一理解）一次性分析**

Prompt（作为文本部分随图片、音频一起传入）：
```
你正在帮一个"演唱会后情绪整理"项目做分析。你会同时收到一张演唱会现场照片
（或视频画面截图）和一段用户跟唱录音（可能混合了视频原声），
请结合两者做综合判断，输出以下三项内容，严格按JSON格式：

{
  "照片氛围描述": "2-3句话，描述画面的色调、光线、人群状态，
                  以及给人的情绪联想，不要以'这张照片'开头，直接进入画面本身",
  "声音情绪描述": "1句话，描述录音中用户的情绪状态和声音表现
                  （比如音量起伏、是否哽咽/嘶吼/轻声跟唱、
                  情绪爆发大致出现在'前段/中段/尾声'），
                  语气偏感性但不要夸张，如果只有视频原声没有独立人声，
                  就基于环境声和人群反应描述现场氛围",
  "情绪标签": "从下面词库中选1个最贴切的：
              怀念、治愈、不舍、热血、孤独中的连接感、
              释放、勇敢、自由、坚持、感动"
}

如果提供了用户文字，也一并纳入判断：{用户文字，可能为空}
只输出JSON，不要有多余解释。
```

### 第二层：认领单生成（等①听歌识曲/备选方案确定具体歌名后触发）

- 输入：OCR演出信息 + 确定的歌曲名（识别成功或备选方案得出的） + 第一层的【照片氛围描述】【声音情绪描述】
- System Prompt：
```
你是"QQ音乐失物招领处"的文案生成AI。你的任务是把用户演唱会当晚的
情绪线索，转化成一张"失物认领单"，语气克制、细腻、略带诗意，
不要煽情过度，不要使用陈词滥调（比如"永远的回忆""青春万岁"）。

你会收到：演出信息、确定的歌曲名、照片氛围描述、声音情绪描述。

请生成以下字段，严格输出JSON：
{
  "遗失物名称": "一个短语，8-16字，描述用户'弄丢的'不是物品，
                而是某种自己的状态/情绪/身份，
                例如'那个敢把心事唱出来的自己'",
  "发现地点": "格式为《歌曲名》+ 一个具体时间点描述，
              时间点用'副歌响起后不久''最后一次合唱的时候'
              这类有画面感但不假装精确的表达",
  "失而复得的感想": "3-4句话，第一人称视角，建议结构：
                    先说'你以为丢失的是XX'，再反转说
                    '你真正丢失/找回的是XX'，最后一句落回
                    '这些情绪其实一直留在音乐里'这个主题，
                    但措辞每次都要不一样",
  "情绪标签": ["复用第一层已生成的情绪标签"],
  "情绪强度": "1-10整数"
}
只输出JSON，不要多余解释。
```

## 三、归途电台 —— 五步的具体API映射

| 步骤 | 内容 | 用什么接口/方式 | 依赖什么 | 何时能开始算 |
|---|---|---|---|---|
| ① 现场的余温 | 识别当晚具体唱的歌 | 听歌识曲API，送入截取后的30秒录音 | 用户上传录音 | 第3页录音上传完立即触发 |
| ② 情绪共鸣的歌 | 相似单曲推荐 | `fcg_music_custom_get_similar_song.fcg`，传入①的song_id | ①完成 | ①返回后立即触发 |
| ③ 观众共同循环的旋律 | 该艺人热门曲目 | `fcg_music_custom_query_singer_list.fcg` + `fcg_music_custom_search.fcg` | 只需要艺人名（OCR结果） | 第2页OCR一出结果立刻后台跑，结果缓存 |
| ④ 许久未听、有不同感受 | 同艺人非当前识别曲目的其他作品 | 复用②或③的结果，过滤掉已用过的曲目 | ②或③任一完成 | 跟随②③自然产出 |
| ⑤ 送你回现实的歌 | 音乐意图处理，情绪匹配 | `music_skill`接口，TrackType传情绪标签 | 情绪标签（第一层豆包调用产出） | 不依赖歌曲识别结果，可与①并行 |

### 备选方案：听歌识曲失败或未识别出歌曲时

用户上传的录音质量不稳定（噪音大、跟唱声音太小、录到的是观众欢呼而非歌曲等），
听歌识曲有一定概率识别失败，这种情况不能让整条链路卡住，需要降级：

1. **①现场余温的替代**：直接使用第2步已缓存的"艺人热门曲目"结果（③的候选列表）里
   热度最高的一首，作为替代锚点，不再声称"识别出你唱的是这首"，
   而是调整措辞为更含糊但依然有诗意的表达，比如"那晚的旋律里，藏着这首歌的回声"
2. **认领单"发现地点"字段联动调整**：不能再写"《具体歌名》副歌响起后第XX秒"，
   要降级为不依赖具体歌名和精确位置的表达，例如"某一段旋律响起、你没忍住跟唱的时候"
3. **②③④⑤后续步骤不受影响**：本来就依赖艺人名/替代曲目而非一定要精确识别结果，
   使用替代锚点后可以正常继续走完整个归途电台的组装逻辑
4. **前端提示**：不需要向用户坦白"识别失败"，走降级方案时前端文案
   应该和正常识别成功时保持一致的诗意语气，用户不会感知到这是"备用方案"

### 归途电台专用的LLM调用（总文案，等②③④⑤四首歌全部就绪后触发）

```
你是"QQ音乐失物招领处"的文案生成AI，现在需要为用户生成一份
"归途电台"的介绍文案。这份电台包含以下五首歌（第一首是识别出的
现场原声或替代锚点，后四首按顺序分别代表：情绪共鸣、观众共同循环、
许久未听有新感受、送你回到现实），请为每首歌写一句推荐语
（10-20字，说明为什么放在这个位置），并写一段50-80字的
总介绍文案，串联起这五首歌的情绪走向，呼应"有些东西没有丢失，
只是暂时留在音乐里"这个主题。

五首歌信息：
1（现场余温）：{①或备选方案的歌名+歌手}
2（情绪共鸣）：{②的歌名+歌手}
3（观众共同循环）：{③的歌名+歌手}
4（许久未听）：{④的歌名+歌手}
5（送你回现实）：{⑤的歌名+歌手}

严格输出JSON：
{
  "总介绍文案": "string",
  "推荐语": ["对应5首歌的推荐语，按顺序排列"]
}
```

## 四、并行策略总结

1. OCR完成 → 立即触发③（艺人热门曲目查询），结果缓存
2. 录音上传完成 → 立即触发①听歌识曲，同时并行触发豆包多模态分析（互不依赖）
3. ①返回（或触发备选方案）后 → 并行触发②相似单曲推荐 + 认领单生成LLM调用
4. ②③④⑤全部就绪后 → 最后触发归途电台总文案LLM调用
5. "先听一首歌"过渡页让等待不再是强制的，即使后台没那么快，用户也有自然的停留理由

## 五、"先听一首歌"过渡页 —— 详细设计说明

这一页承接在原有搜索动画之后，是整个体验里**情绪最沉浸的一个停顿点**——不是"加载中"的技术性等待，而是让用户在这几十秒里，真正安静地重新回到"刚看完演唱会"的那种恍惚状态。

### 视觉基调

延续项目已有的暗色复古美学（认领单那种做旧纸张、打字机质感的路线），但这一页要更**沉一点、暗一点**，像是走进了失物招领处后台那个只属于深夜的小房间：

- **背景**：深棕/深灰渐变，边缘略带暗角（vignette）效果，中央留出一小片暖光，像一盏老式台灯照在唱片机上
- **核心视觉元素**：一张缓慢旋转的黑胶唱片（CSS动画即可实现，转速要慢，营造"时间被拉长"的感觉，不是DJ打碟那种快速旋转），唱针轻轻搭在上面
- **文字排版**：全程用衬线字体或手写体感的字型（和认领单的"失物认领单"标题风格呼应），文字量要克制，不堆砌

### 页面文案（可直接使用，也可以按你的语气微调）

页面刚进入时，唱片开始转动，逐字浮现一行小字：
```
在电台准备好之前，先请你听一首歌。
```

稍作停顿后，副标题浮现：
```
这段安静，也是属于那个夜晚的一部分。
```

如果①已识别出具体歌曲，播放的就是这首歌的试听片段，同时显示：
```
正在播放
《{识别出的歌名}》— {歌手}
```

如果①还未识别完成，播放预设的"入场氛围"备选曲目（提前准备3-5首风格统一、适合安静聆听的曲子），显示：
```
正在播放
一段属于散场后的旋律
```
（不显示具体歌名，避免和之后可能识别出的真实歌曲产生矛盾）

### 按钮的状态设计

页面底部有一个按钮，是整个页面**唯一的交互点**，三种状态：

**状态一（后台数据还在处理中）**：
按钮呈半透明/灰色调，不可点击，文字缓慢呼吸闪烁：
```
电台整理中...
```

**状态二（后台数据全部就绪）**：
按钮转为实色（可以用认领单上那个红色"FOUND"印章的红作为强调色），
出现一个轻微的"落定"动效（比如轻微放大再回弹一下，像盖章的感觉），文字变为：
```
准备好了，前往归途电台
```

**状态三（超过20秒仍未就绪，兜底）**：
```
遇到点小问题，先看看已经准备好的内容
```
点击后用已完成的部分内容（哪怕只有①和情绪标签）继续往下走，不强行卡住用户。

### 交互细节

- 用户不需要等按钮变化才能操作——如果他们此刻就想点击，允许他们在"电台整理中..."状态下也能点一次"催一下"（可选，非必须），但更推荐的做法是让唱片旋转和文字呼吸感本身就足够让人愿意多停留几秒，而不是让用户觉得"这里被卡住了"
- 音量条／播放进度条可以做得极简，一条纤细的横线，不需要花哨的可视化频谱
- 这一页停留的时间没有下限要求，用户可以听完一整首试听片段再点，也可以数据一就绪就立刻离开——这是"用户自主选择要不要多留一会"，而不是"必须等这么久"

## 六、豆包API调用说明

### 请求结构（以图片+音频+文本同一次调用为例）

音频部分需要转成base64，并指定`format`字段（对应支持的MIME类型，比如mp3对应`audio/mpeg`）。图片可以用URL或base64。整体建议使用Chat Completions风格的多模态调用（`messages.content`数组里混合`type: "image_url"` / `type: "input_audio"` / `type: "text"`），而不是`/responses`接口示例里那种简化的单图片写法——因为音频输入的字段结构（`input_audio.data` + `input_audio.format`）目前有官方文档支持的是Chat API路径，请Trae在实现时对照火山方舟官方文档确认`/responses`接口是否已支持同等的音频字段，如果没有就统一走Chat Completions的调用方式。

### 关键限制（实现时要注意）
- 音频文件大小上限25MB（压缩到64kbps单声道后完全没问题，一般几十秒录音也就几百KB）
- 单次请求音频总时长上限120分钟（远超我们的使用场景，不用担心）
- API Key只放在后端环境变量里，绝不能出现在前端代码或提交到Git的文件中

## 七、给Trae的完整指令（可直接复制）

```
我需要把项目里的假数据替换成真实功能，具体分两大块：认领单生成、归途电台生成，
并新增一个"先听一首歌"的过渡页。请按以下顺序实现，所有涉及API Key的调用
都通过后端API Route转发，不能在前端直接暴露密钥。

【第1步：票根OCR + 艺人信息提前查询】
用户上传票根图片后：
1. 创建/api/ocr接口，调用百度OCR识别文字，用LLM结构化提取
   {演出名称、艺人、日期、城市、场馆}
2. OCR成功拿到艺人名后，立即后台并行发起：
   - fcg_music_custom_query_singer_list.fcg 确认歌手singer_mid
   - fcg_music_custom_search.fcg 按该艺人名搜索，按热度取前10首
   结果缓存在当前用户会话状态里

【第2步：视觉素材准备】
1. 用户上传照片，保留原图
2. 用户上传视频时，前端用<video>+<canvas>方案，在视频40%-60%时长位置
   抽取一帧，转成JPEG（压缩质量0.8），作为"照片"素材

【第3步：音频素材准备】
1. 用户录音完成后，压缩为64kbps左右单声道mp3
2. 如果用户上传了视频，提取视频音轨，与录音合并为同一份音频素材
   （不要分别处理再自己合并结果）
3. 未压缩的原始录音前8-10秒单独送去听歌识曲API识别具体歌曲，
   这一步和音频压缩处理并行进行

【第4步：豆包多模态分析（合并照片+音频+文字为一次调用）】
创建/api/analyze-multimodal接口，把第2步的图片素材、第3步压缩后的音频素材、
